# Factory Data

The Matter&trade; standard requires device specific data to be stored in program flash for use at runtime.
This data is computed at the factory line and flashed together with the firmware image.

To facilitate this process the factory block was created, a simple data structure based on the [TLV](#data-format) (type/length/value) concept.
No CRC or other content verification is done at runtime.

The factory block can store:

* [developer-only] DAC certificate (device attestation certificate)
* [developer-only] DAC key
* [developer-only] PAI certificate (product attestation certificate)
* CD certificate (certificate declaration)
* spake2p items (passcode, #iterations, salt)
* A unique ID (Rotating device ID unique ID)
* customer specific data if any free space is available

The factory block flash region is only writable during production line flashing, it can not be changed later on.

## Implementation overview

The implementation consists of 4 parts:

* A data-format as means of data-exchange between factory line and firmware.
* A python tool to generate the binary data section content.
* C code to look up the stored data at runtime.
* A Makefile to facilitate 'integration into development flow' by re-generating the factory data payload before the firmware linking step.

## Data format

The file starts with a header word reading 'QFDA' (for Qorvo&reg; factory data in ASCII).

After this, the data is stored encoded as a simple type/length/data (TLV) format.
This means for each element stored, the following three items are stored:
* A type word (4 - bytes), identifying what kind of data is to follow
* A length word (4 - bytes) indicating the length of the data that will follow
* A variable length Value.

The last TLV element is the 'End Marker'. It is an element of type 0 and length 0, stored as [0x00, 0x00].

Every TLV element is word-aligned. If necessary - add padding bytes (0x00) after the variable length value to align the type word correctly to a 32-bit word boundary.

### Arguments (configuration) file

When argument list is particularly long like for this tool, it is recommended to keep the list of arguments in a (configuration) file rather than typing it out at the command line.
For this reason, arguments that start with an '@' character will be treated as a 'read this file'-statement, and @ arguments will be replaced by the argument list read from the file. For example:

```shell
cat factory_data.conf
--discriminator=3840
--passcode=20202021
python generate_factory_data.py --discriminator=1234 @factory_data.conf --out_file=data.bin
```

Arguments read from a file must by default be one per line and are treated as if they were in the same place as the original file referencing argument on the command line.
The above example, resulting in arguments set to: "--discriminator=1234 --discriminator=3840 --passcode=20202021 --out\_file=data.bin"

## Example

This example creates a factory data file with two application specific elements, with manufacturer specific tag id's 0x40000000 and 0x40000001.
Two strings are used, the first is 3 bytes long, the second one 4 bytes long.

```shell
echo -n 'one' >/tmp/one #(will not add a newline)
echo 'two' >/tmp/two    #(will add a newline)
python generate_factory_data.py --out_file /tmp/example.bin --data 0x40000000:@/tmp/one --data 0x40000001:@/tmp/two
```

## Firmware data

The factory data generator mechanism is also used to embed binary data into the upgradable part of the firmware image. Since the Matter Certificate Declaration is tied to a firmware version, it needs to be updated along with the software.

It typically contains:

* CD certificate (certificate declaration)
* customer specific data if needed
