# Application Bootloader

The Application Bootloader is an application itself. That means it runs in the application space of the QPG6200. It assists in launching other applications:

* manage execution of two different applications (primary and secondary) - see **[Bootloader Metadata](#terms-and-concepts)** and **[Dual Boot](#dual-mode-application-bootloader)** section.
* implement an **[Over-The-Air (OTA)](#ota-upgrade-mechanism)** upgrade mechanism. This allows secure and verified application deployments without physical access to the device.

This chapter describes the architecture of the Application Bootloader and its functionality.

## Variants

Application Bootloader has 2 variants:

* `Bootloader` - launch a single application from device memory every time the device boots. See: **[Single mode Application Bootloader](#single-mode-application-bootloader)**
* `Bootloader_dualboot` - choose and launch 1 of 2 available applications from memory on every boot. Those are called `Combo` applications. See: **[Dual mode Application Bootloader](#dual-mode-application-bootloader)**

## Functionality
### Terms and Concepts

**Bootloader Metadata**: Used for OTA updates and booting the application. It is an 8-byte area at address 0x1001f000. It stores two addresses in device RRAM space:
The bootloader metadata are generated by the gen_bl_metadata.py tool that is available for in "Tools/Bootloader/". For more information about generation details refer to [Bootloader metadata generator](../../Tools/Bootloader/README.md)

| Offset | Size    | field description                               |
|--------|---------|-------------------------------------------------|
| 0      | 4 bytes | Address of OTA update image                     |
| 4      | 4 bytes | Address of application to launch on device boot |

When using `Bootloader_dualboot` this area can be modified by an application. This allows the `Bootloader_dualboot` to switch between applications and perform OTA update.

**OTA update**: Over-The-Air update. This is performed by the Application Bootloader using a compressed update image placed on the device via the Matter protocol.

**Secure Boot**: Protects the device against execution of unauthorized code. It uses the Secure Element to verify applications with a Manufacturer Public Key. This key is permanently stored on the device. Configuration of Secure Boot and Manufacturer Public Key happens during device manufacturing process.

**Manufacturer Public Key**: Used in **[Secure Boot mode](#terms-and-concepts)** to cryptographically verify code before execution. It is saved in the secure storage eOTP area during manufacturing process. See: [Keys generation and conversion](https://qorvo_sdk.gitlab.io/public/tools/packages/qorvo-platform-tools/security_toolbox/commands.html#keys-generation-and-conversion)

**Developer Public Key**: Stored in the **[Application/OTA Header](#terms-and-concepts)**. Used to verify the application's digital signature. See: [Keys generation and conversion](https://qorvo_sdk.gitlab.io/public/tools/packages/qorvo-platform-tools/security_toolbox/commands.html#keys-generation-and-conversion)

**Application/OTA Header**: Header required by the Application Bootloader. Applications and OTA update images must begin with this header. It is used for cryptographic verification and configurations. The header must not be compressed. It is 512 bytes of data:

| Offset | Size       | field description                                                             |
|--------|------------|-------------------------------------------------------------------------------|
| 0x00   | 0xF4 bytes | Developer Certificate and Developer Public Key                                |
| 0x0F4  | 0x40 bytes | P-256 Signature over Authentication Header, Version Info, reserved and binary |
| 0x134  | 0x40 bytes | Authenticated Header                                                          |
| 0x174  | 0x10 bytes | Version Info                                                                  |
| 0x184  | 0x70 bytes | reserved                                                                      |
| 0x1F4  | 0x0C bytes | padding with 0x00                                                             |

### Single mode Application Bootloader

In Single Boot mode, the Application Bootloader always loads an application from the address listed in **[Bootloader Metadata](#terms-and-concepts)**.

### Dual mode Application Bootloader

In Dual Boot mode the **application** controls both address values in **[Bootloader Metadata](#terms-and-concepts)**:
- It can change the address of the application. That application will be loaded after reboot.
- It can change the address for OTA update image. That address will be checked for update after reboot.

Dual Boot acts as a **[Fallback mechanism](#fallback-mechanism)** when an application fails to load.

*Usage example*:
Assume we have two applications: A and B. Both have their unique addresses. Application A has its address listed in **[Bootloader Metadata](#terms-and-concepts)**. Upon boot, application A loads.

To boot application B:
* Application A must change the address in **[Bootloader Metadata](#terms-and-concepts)** to point to address of application B.
* The device must undergo a reboot. This can be invoked by application A, or through other means.
* On boot, the device finds entry in **[Bootloader Metadata](#terms-and-concepts)** and boots application B.

#### OTA Upgrade Mechanism

On boot, the Application Bootloader first loads the addresses from **[Bootloader Metadata](#terms-and-concepts)**: first the OTA update image address, then the address of application to launch. Next:

1. The Application Bootloader retrieves both **[Headers](#terms-and-concepts)**:

    - for the OTA update image
    - for the current application.

2. The Application Bootloader compares the headers (see: **[Application/OTA Header](#terms-and-concepts)** -> Version Info). If version number in the OTA update image is higher than current application header, Application Bootloader initiates the update process.

    - **NOTE** 1: It is impossible to downgrade the image using this method
    - **NOTE** 2: If both versions are equal, the OTA update image is skipped.

3. The bootloader initiates the OTA verification procedure:

    - Scenario 1 - **[Secure Boot](#terms-and-concepts)** enabled: The Application Bootloader queries the Secure Element to check if the **[Developer Public Key and Certificate](#terms-and-concepts)** verify against the **[Manufacturer Public Key](#terms-and-concepts)**. If `TRUE`, the update proceeds.
    - Scenario 2 - Normal operation: The Application Bootloader checks the digital signature embedded in the **[OTA Header](#terms-and-concepts)** with **[Developer Public Key](#terms-and-concepts)** from developer certificate to verify digest of OTA image. This aims to test the image integrity. If correct, the update proceeds.

4. The Application Bootloader unpacks the OTA image using an LZMA algorithm directly into the application address space, starting with the address specified in the header.

    - **NOTE**: The compressed LZMA payload stores a compressed **[Application Header](#terms-and-concepts)**. This header becomes the new application header and is verified every time the application launches.
    - **WARNING**: This operation is unbuffered, and directly modifies existing application memory space. When interrupted, it results in an unbootable state.

5. The Application Bootloader reboots the device and begins at step 1. As a result:

    - the OTA update image should be deleted
    - updated application should boot

**NOTE**: OTA update image is placed on the device using Matter. Application Bootloader does not control that.

**WARNING**: OTA image size has a limit. If unpacked image size is too big, it can overwrite data in adjacent memory sector. No protection against this is implemented currently.

#### Application Launch Process

On boot, the Application Bootloader loads addresses from **[Bootloader Metadata](#terms-and-concepts)** and compares their headers. If there are no conditions for an **[OTA update](#ota-upgrade-mechanism)**, the Application Bootloader proceeds to load an application specified in Metadata address.

This process begins with verification of the application. It happens on every boot, and is identical to the process of **[OTA update](#ota-upgrade-mechanism)**:

   - Scenario 1 - **[Secure Boot](#terms-and-concepts)** enabled: The Application Bootloader queries the Secure Element to check if the **[Developer Public Key and Certificate](#terms-and-concepts)** in the **[Application Header](#terms-and-concepts)** verifies against the **[Manufacturer Public Key](#terms-and-concepts)**. If `TRUE`, the application launch proceeds.
   - Scenario 2 - Normal operation: The Application Bootloader checks the digital signature embedded in the **[Application Header](#terms-and-concepts)** with **[Developer Public Key](#terms-and-concepts)** from developer certificate to verify digest of the application. This aims to test the application integrity. If correct, the application launch proceeds.

If these verifications are successful, the bootloader sets the vector table of the application and jumps to the boot address of the selected application.

**WARNING**: No memory protection mechanism is currently implemented. If the application memory space is modified after successful verification, this will not be detected, and the modified application may be executed.

### Fallback Mechanism
If the application passes verification, but the launch process fails, the Application Bootloader will attempt to run the primary application.
This is designed to protect against scenarios where:
   - application B is set as active in the **[Metadata](#terms-and-concepts)** but it fails to boot,
   - temporary error prevents application A from launching correctly on the first attempt.
This does not protect against errors resulting from physical damage or flash memory wear. These may cause a boot loop.

## Notes
1. There are 2 makefiles in the bootloader folder:
   - Makefile.Bootloader_qpg6200_compr_secure_dualboot - for compiling and generating a compressed secure dualboot bootloader binary
   - Makefile.Bootloader_qpg6200_compr_signed - for compiling and generating a compressed signed bootloader binary, e.g. the current Zigbee Light application.

   You can use Makefile.Bootloader_qpg6200_compr_secure_dualboot to build the application bootloader binary. You can find more details in [Installation](../../sphinx/docs/QUICKSTART.rst#install-the-sdk) and [Working with examples](../../sphinx/docs/QUICKSTART.rst#working-with-examples) guides.
   ```
   cd Applications/Bootloader
   make -f Makefile.Bootloader_qpg6200_compr_secure_dualboot clean
   make -f Makefile.Bootloader_qpg6200_compr_secure_dualboot
   ```
2. These are limitations in the current release:
   - OTA applications should always be compressed
